# 作業手順

## 進め方（TDD）

機能ごとに以下を繰り返す：

1. テストコードを書く
2. 機能を実装する
3. テストが通るか確認する
4. ドキュメントを更新する（`design.md`・`README.md`）
5. コミットする

### ドキュメント更新の対象

| ドキュメント | 更新するタイミング |
|-------------|------------------|
| `design.md` | 機能の実装状況・ファイル構成が変わったとき |
| `README.md` | 利用者向けの使い方が変わったとき |
| `procedure.md` | 作業手順・方針が変わったとき |

### テストの種類

各機能に対して2種類のテストを書く：

| 種類 | 対象 | ファイル例 |
|------|------|-----------|
| ロジックテスト | `app.js` の関数（データ操作） | `test_edit.html` |
| UIテスト | DOM の動作（クリック・入力・表示） | `test_edit_ui.html` |

**ロジックテストの例：** `updateMemo()` を呼んだ結果を確認する  
**UIテストの例：** ボタンをクリックしたら編集エリアが開くか確認する

## 実装する機能の順番

1. メモの新規作成
2. メモの一覧表示
3. メモの編集・保存（localStorage）
4. メモの削除

## 新機能追加チェックリスト

機能追加時は以下の順番で進める：

- [ ] テストコードを書く（ロジック・UIの両方）
- [ ] 機能を実装する
- [ ] テストが通ることを確認する
- [ ] `index.html` をブラウザで開いて動作確認する
- [ ] リグレッションテストを実行する（`test/test.html`）
- [ ] ドキュメントを更新する（`design.md`・必要なら `README.md`）
- [ ] コミットする

## ブランチ運用

機能追加や修正はブランチで作業し、完成後に `master` にマージする。

```powershell
# 機能ブランチを作成
git checkout -b feature/○○

# 作業・コミットを繰り返す

# master にマージ
git checkout master
git merge feature/○○

# ブランチを削除
git branch -d feature/○○
```

### ブランチ名の例

| 種類 | 例 |
|------|-----|
| 新機能 | `feature/responsive` |
| バグ修正 | `fix/delete-button` |
| リファクタリング | `refactor/test-cases` |

## コミット前の確認

コミットする前に必ず以下を確認する：

1. `test/test.html` を開いて全テストが通ることを確認する
2. `index.html` をブラウザで開いて手動動作確認をする
3. `git status` で意図しないファイルが含まれていないか確認する

## テストが失敗したときの切り分け手順

1. `test/test.html` で失敗しているテスト名を確認する
2. 該当する standalone テストファイルを開いて詳細を確認する
3. 失敗しているのがロジックかUIか判断する
   - ロジックテスト失敗 → `app.js` を調べる
   - UIテスト失敗 → `ui.js` または `index.html` を調べる
4. 直近のコミットで何を変更したか `git log` で確認する
5. 修正してテストを再実行する

## コードレビューの観点

実装後に以下の点を自分でチェックする：

| 観点 | 確認内容 |
|------|---------|
| 正確性 | 意図した通りに動くか |
| 一貫性 | 既存のコードのスタイルに合っているか |
| 安全性 | エラー処理が必要な箇所に抜けがないか |
| 重複 | 同じ処理が複数箇所に書かれていないか |
| テスト | 新しいコードに対応するテストがあるか |

## 機能追加後の動作確認手順

`index.html` を開いて以下を確認する：

1. 新機能が正しく動くか（ゴールデンパス）
2. 既存機能が壊れていないか
   - メモの新規作成
   - メモの選択・編集
   - メモの削除
   - ページを再読み込みしてもデータが残るか

## テストの妥当性監査

テストコードが正しく機能を網羅しているか定期的に確認する。

### 確認するタイミング

- 複数の機能が揃ったとき
- 新機能を追加する前
- バグが多発したとき

### 監査の観点

| 観点 | 確認内容 |
|------|---------|
| 網羅性 | ロジックテストとUIテストの両方があるか |
| 独立性 | テスト間で状態が汚染されていないか（`setup()` の後処理は十分か） |
| 境界ケース | 存在しない id・空文字・大量データなど異常系が含まれているか |
| 実態との一致 | テストが実際のコードの動作と合っているか |

### 監査で問題が見つかった場合

1. 不足テストを追加する
2. 誤ったテストを修正する
3. リグレッションテストを実行して全体を確認する

## リグレッションテスト

機能を追加・修正したあと、既存の機能が壊れていないか確認する。

### 手順

1. `test/test.html` を開く
2. 全テストファイルのリンクを順番に開いて確認する
3. すべて成功していればコミットする
4. 失敗があれば修正してから再確認する

### 確認するタイミング

- 新機能を実装したとき
- バグを修正したとき
- ファイル構成を変更したとき（リファクタリング）

## git による変更記録

### 初回のみ（リポジトリの初期化）

```powershell
git init
git add .
git commit -m "初期設定：設計書と作業手順を追加"
```

### 機能ごとのコミット手順

```powershell
# テストを書いたとき
git add .
git commit -m "test: ○○のテストを追加"

# 機能を実装したとき
git add .
git commit -m "feat: ○○を実装"

# バグを修正したとき
git add .
git commit -m "fix: ○○を修正"
```

### コミットメッセージの prefix

| prefix | 意味 |
|--------|------|
| `feat:` | 新機能 |
| `test:` | テスト |
| `fix:` | バグ修正 |
| `docs:` | ドキュメント |

## Claude との作業再開方法

### このセッションを続ける場合
- そのままチャットで指示するだけでOK

### 新しいセッションを始める場合
- Claude は前の会話を覚えていないため、以下のように伝える

```
design.md と procedure.md を読んで、続きを手伝って
```

- Claude がファイルを読んで現在の状態と次のタスクを把握する

### 作業再開前に確認すること

1. `git log --oneline` で直近のコミットを確認する
2. `git status` で未コミットの変更がないか確認する
3. `test/test.html` を開いて全テストが通ることを確認する
4. `design.md` の機能一覧で次にやることを確認する

### やり直し・修正したい場合
- コード修正：チャットで「○○を変えて」と指示する
- 特定時点に戻す：`git checkout` や `git revert` を使う
- コミットをこまめにしておくと安全に巻き戻せる

## test.html の動作説明

### 画面に表示されるもの

- 各テストの結果が1行ずつ表示される
  - 成功：`✓ テスト名`（緑）
  - 失敗：`✗ テスト名 — エラー内容`（赤）
- 最後に合計結果が表示される（例：`5 テスト中 5 件成功、0 件失敗`）

### 内部の動き

1. `createMemo()` などのテスト対象関数を呼び出す
2. 各 `test()` が `assert()` で期待値をチェックする
3. `assert()` の条件が false なら例外を投げ、失敗として記録する

### 注意点

- 現時点では `createMemo()` は `test.html` 内に直接記述している
- `index.html` に実装後は、そちらの関数と差し替える
